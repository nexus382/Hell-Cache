Performance Limits & Constraints

Aspect Limit/Constraint Impact
Resolution Fixed at 240×240 pixels Cannot change resolution, all content must fit
Color Depth 16-bit RGB565 only Limited color palette (65,536 colors), banding possible
Double Buffering Supported but uses extra memory Back buffer = 115KB additional RAM
Sprite Memory Stored in C memory space Efficient, but must free manually
No Hardware Acceleration All rendering is software-based CPU-intensive, must optimize draw calls

Performance Critical Constraints

  • Maximum 60 FPS achievable only with optimized rendering
  • Complex scenes may drop to 30-45 FPS without optimization
  • Alpha blending and blur effects are expensive operations
  • Overdraw (drawing same pixels multiple times) kills performance
  • Sprite scaling is CPU-intensive (especially non-uniform scaling)
  • Mosaic effects have moderate performance cost

Best Practices

Rendering Optimization Tips

  • Use double buffering to eliminate flicker
  • Minimize overdraw by drawing back-to-front
  • Avoid per-pixel operations when possible
  • Use sprite scaling sparingly (cache scaled versions if frequent)
  • Limit blur radius (0-3 is subtle, 8-10 is extreme)
  • Batch similar drawing operations together
  • Use Z-index to minimize redundant draw calls
  • Consider lower FPS targets (30-45 FPS) for complex scenes
  • Profile performance with frame time monitoring

Color Format Guide

RGB565 Format: Red (5 bits) + Green (6 bits) + Blue (5 bits)

-- Color conversion helper
function rgb_to_565(r, g, b)
    local r5 = math.floor((r * 31) / 255)
    local g6 = math.floor((g * 63) / 255)
    local b5 = math.floor((b * 31) / 255)
    return (r5 << 11) | (g6 << 5) | b5
end

Critical Audio Requirements

  • MUST call vmupro.sound.update() every frame or no audio plays
  • MUST call audio.startListenMode() before using audio features
  • MUST call audio.exitListenMode() when done with audio (e.g., screen change)
  • Forgetting lifecycle management causes memory leaks and audio failures
  • Infinite looping not implemented (-1), use high repeat count (99) instead

Input Detection Methods

Edge Detection

  • pressed() - Button just went down this frame
  • released() - Button just went up this frame
  • Ideal for menu navigation
  • Prevents repeat triggers

State Detection

  • held() - Button is currently down
  • anythingHeld() - Any button is down
  • Ideal for continuous movement
  • Good for charging attacks

Convenience Functions

  • confirmPressed() - A button edge
  • confirmReleased() - A button release
  • dismissPressed() - B button edge
  • dismissReleased() - B button release
  • UI-focused helpers

Input Requirements

Critical Input Requirements

  • MUST call vmupro.input.read() ONCE per frame before checking buttons
  • Calling read() multiple times per frame will miss edge detections
  • Not calling read() means stale button state
  • Edge detection (pressed/released) only works if read() called each frame

Input Best Practices

  • Call input.read() once at start of update loop
  • Use pressed() for menu navigation and one-shot actions
  • Use held() for continuous movement and charging
  • Implement deadzone for analog-like control (if needed)
  • Buffer rapid inputs to prevent accidental triggers
  • Use confirmPressed/dismissPressed for UI consistency

Code Examples

function vmupro.update()
    -- CRITICAL: Read input once per frame
    vmupro.input.read()

    -- One-shot actions (edge detection)
    if vmupro.input.pressed(vmupro.input.A) then
        player_jump()
    end

    -- Continuous actions (state detection)
    if vmupro.input.held(vmupro.input.LEFT) then
        player.x = player.x - player.speed
    end

    if vmupro.input.held(vmupro.input.RIGHT) then
        player.x = player.x + player.speed
    end

    -- Menu navigation
    if vmupro.input.pressed(vmupro.input.UP) then
        menu_index = menu_index - 1
    end

    if vmupro.input.pressed(vmupro.input.DOWN) then
        menu_index = menu_index + 1
    end

    -- UI helpers
    if vmupro.input.confirmPressed() then
        activate_menu_item()
    end

    if vmupro.input.dismissPressed() then
        close_menu()
    end
end

4. Text & UI Capabilities

Text Rendering Features

Core Features

  • Built-in bitmap font
  • Arbitrary text string rendering
  • Custom foreground color (RGB565)
  • Custom background color (RGB565)
  • Pixel-level positioning
  • No text wrapping (manual)
  • No font selection (fixed system font)

Limitations

  • No word/line wrapping automation
  • No custom font loading
  • No text scaling
  • No text alignment options
  • No text effects (shadows, outlines)
  • Fixed character size (8x8 pixels typically)
  • Limited character set (ASCII-focused)

UI Building

  • Manual text positioning
  • Background color for text boxes
  • Primitive-based UI (rectangles, lines)
  • Sprite-based UI elements
  • Custom font rendering via sprites (if needed)

Text Best Practices

UI Implementation Tips

  • Use sprites for complex text (custom fonts, styled text)
  • Implement manual word wrapping for dialog boxes
  • Cache common text strings as sprites for performance
  • Use draw primitives for UI frames and borders
  • Layer UI elements with Z-index for proper overlap
  • Create text box helper functions for consistency
  • Pre-render static UI to offscreen buffer if possible

Text Limitations

  • Cannot load custom TTF/OTF fonts (use sprite fonts instead)
  • No advanced text rendering (shadows, gradients, warping)
  • Fixed-width characters only (no proportional spacing adjustment)
  • Extended characters may have limited support
  • Text rendering is CPU-intensive (cache as sprites when possible)

Code Examples

-- Text rendering
vmupro.graphics.drawText("Score: 100", 10, 10, vmupro.graphics.WHITE, vmupro.graphics.BLACK)

-- Text box helper
function draw_text_box(text, x, y, width, height, fg_color, bg_color)
    -- Draw box background
    vmupro.graphics.drawFillRect(x, y, x + width, y + height, bg_color)

    -- Draw text
    vmupro.graphics.drawText(text, x + 4, y + 4, fg_color, bg_color)

    -- Draw border
    vmupro.graphics.drawRect(x, y, x + width, y + height, fg_color)
end

-- Manual word wrapping
function draw_wrapped_text(text, x, y, max_width, fg, bg)
    local words = {}
    for word in string.gmatch(text, "[^%s]+") do
        table.insert(words, word)
    end

    local line = ""
    local current_y = y
    local char_width = 8  -- Assume 8x8 font

    for i, word in ipairs(words) do
        local test_line = line == "" and word or line .. " " .. word
        if #test_line * char_width > max_width then
            vmupro.graphics.drawText(line, x, current_y, fg, bg)
            current_y = current_y + 10
            line = word
        else
            line = test_line
        end
    end

    -- Draw last line
    vmupro.graphics.drawText(line, x, current_y, fg, bg)
end

Code Examples

-- 60 FPS game loop with timing
local frame_time_target = 16667  -- 60 FPS in microseconds

function game_loop()
    local frame_start = vmupro.system.getTimeUs()

    -- Update game logic
    update_game()

    -- Render frame
    render_frame()
    vmupro.graphics.pushDoubleBufferFrame()

    -- Calculate frame time
    local frame_time = vmupro.system.getTimeUs() - frame_start

    -- Sleep to maintain 60 FPS
    if frame_time < frame_time_target then
        vmupro.system.delayUs(frame_time_target - frame_time)
    else
        vmupro.system.log(vmupro.system.LOG_WARN, "Performance",
            "Frame exceeded budget: " .. frame_time .. "us")
    end
end

-- Memory check before loading large asset
local required_bytes = 50000  -- 50KB
local largest_free = vmupro.system.getLargestFreeBlock()

if largest_free >= required_bytes then
    local music = vmupro.sound.sample.new("music/large_track")
else
    vmupro.system.log(vmupro.system.LOG_ERROR, "Memory",
        "Cannot allocate 50KB, largest free: " .. largest_free)
end

6. File I/O Capabilities

File System Features

File Operations

  • read() - Load entire file into memory
  • write() - Write data to file
  • exists() - Check file existence
  • getSize() - Get file size in bytes
  • deleteFile() - Remove file
  • String-based file contents
  • Binary data support

Folder Operations

  • folderExists() - Check folder existence
  • createFolder() - Create new folder
  • deleteFolder() - Remove empty folder only
  • Must delete files before deleting folder

Helper Functions

  • createFile() - Create empty file
  • Returns true if file exists or created
  • Useful before write() operations
  • No file listing/traversal functions
  • No file metadata (timestamp, permissions)

File System Constraints

Critical File Limitations

  • SD CARD ONLY - Cannot access system files or firmware
  • /sdcard PREFIX REQUIRED - All paths must start with /sdcard/
  • NO DIRECTORY LISTING - Cannot enumerate files in folders
  • NO FILE METADATA - No timestamps, permissions, or attributes
  • ENTIRE FILE LOADED - No streaming or chunked reading
  • MUST EXIST FIRST - createFile() before write() recommended
  • FOLDER MUST BE EMPTY - Cannot delete non-empty folders
  • NO ABSOLUTE PATHS - All paths are /sdcard/ relative

Save Game Implementation Pattern

Since you cannot list directory contents, implement versioned saves or fixed-slot saves:

Code Examples

-- Save game data
function save_game(slot, score, level)
    local filename = "/sdcard/saves/save" .. slot .. ".dat"

    -- Ensure file exists
    vmupro.file.createFile(filename)

    -- Write data as string
    local data = string.format("%d,%d", score, level)
    local success = vmupro.file.write(filename, data)

    if success then
        vmupro.system.log(vmupro.system.LOG_INFO, "Save",
            "Saved slot " .. slot .. ": " .. data)
    else
        vmupro.system.log(vmupro.system.LOG_ERROR, "Save",
            "Failed to write " .. filename)
    end
end

-- Load game data
function load_game(slot)
    local filename = "/sdcard/saves/save" .. slot .. ".dat"

    if vmupro.file.exists(filename) then
        local data = vmupro.file.read(filename)

        -- Parse saved data
        local score, level = string.match(data, "(%d+),(%d+)")

        return {
            score = tonumber(score),
            level = tonumber(level)
        }
    else
        vmupro.system.log(vmupro.system.LOG_WARN, "Save",
            "No save file: " .. filename)
        return nil
    end
end

-- Manage save folders
function init_save_system()
    if not vmupro.file.folderExists("/sdcard/saves") then
        vmupro.file.createFolder("/sdcard/saves")
    end
end

-- Delete save (must delete file before folder can be removed)
function delete_save(slot)
    local filename = "/sdcard/saves/save" .. slot .. ".dat"
    vmupro.file.deleteFile(filename)
end

-- Versioned saves (bypass directory listing limitation)
function get_next_save_slot()
    for slot = 1, 10 do
        local filename = "/sdcard/saves/save" .. slot .. ".dat"
        if not vmupro.file.exists(filename) then
            return slot  -- Found empty slot
        end
    end
    end
    return nil  -- All slots full
end

Recommendations by Genre

Best-Fit Genres

  • Platformers - Use tile-based backgrounds, limit parallax layers to 2-3
  • Shoot 'em Ups - Perfect fit, focus on bullet patterns not count
  • Puzzle Games - Ideal for hardware, low sprite count, focus on UI
  • Top-Down RPGs - Tile-based worlds work well, manage sprite count carefully

Challenging Genres

  • Real-Time Strategy - Too many units, pathfinding CPU issues
  • Fighting Games - Limited to 2 characters, animation complexity constrained
  • First-Person - Raycasting possible but expensive, limited visual fidelity
  • Open World - Memory constraints limit world size, streaming not possible

Not Recommended

  • Complex 3D - No hardware acceleration, software rendering too slow
  • High-Entity Count Games - 240×240 limits onscreen complexity
  • Text-Heavy Adventures - System font limited, use sprite fonts instead
  • Multiplayer - No networking, split-screen impractical
  • Video Playback - No video codec support
